Project Motivation
Problem & Solution
Problem
  • Travel booking requires juggling multiple providers
  • No single offline system handles flights, hotels & tours together
  • Enterprise platforms often present significant overhead for localized solutions
Solution: SmartGo
  • Single Java Swing desktop application
  • Book flights, hotels & tour plans in one session
  • File-based persistence (no database required)
  • Clean OOP architecture showcasing all four pillars
Goal: Deliver a robust application utilizing Encapsulation, Inheritance, Abstraction & Polymorphism within a scalable production architecture.

Architectural Implementation
Design Principles in Practice
1
Encapsulation
  • All 19 models use private fields + public getters/setters
  • Bill fee calculation sealed inside BookingService; GUI never touches it
  • DataStore I/O is private static, so services call exposed API only
2
Inheritance
  • Person → User & Admin: shares id, name, email, phone, password
  • Transport → Flight, Bus, Train: shares price, capacity, destinationId
  • Subclasses call super() to pass common fields to parent
3
Abstraction
  • abstract Person declares getRole() and cannot be instantiated
  • abstract Transport declares getScheduleSummary(), forcing a concrete implementation
  • Service layer hides all file I/O details from the GUI layer
4
Polymorphism
  • getRole() overridden: User returns "User", Admin returns "Admin"
  • getScheduleSummary() returns different string per transport type
  • ReviewService dispatches to flight/hotel/tour target at runtime via type field

AuthService.java: Honest Assessment
Authentication & User Management
What Works
  • Email uniqueness check on registration
  • Session management via loggedInUser static field
  • isLoggedIn() guards all user-only actions
  • Login throws SmartGoException on bad credentials
  • ID continuity on startup using max existing ID + 1
OOP Highlight
User and Admin both extend abstract Person. Shared fields are inherited, getRole() is overridden differently in each. Polymorphism + Inheritance in practice.
⚠️ Acknowledged Limitation: Password Storage
Passwords are stored as plain text in users.txt. The code comment explicitly acknowledges this:
// In a real app you would hash the password
// properly. For this project we store it as-is
// to keep things simple.
User newUser = new User(nextUserId++, name,
  email, phone, password, createdAt, addr, ...);
In production: use BCrypt / PBKDF2 hashing before saving. The field is already named passwordHash in Person.java, so it is ready for an upgrade.

Data Layer
DataStore.java: Single I/O Choke-Point
Three Core Methods
appendLine(file, data)
Adds new record. O(1) write. FileWriter(append=true) so no full rewrite is needed.
readLines(file)
Returns List<String>. Skips blank lines. Splits on "|" delimiter. Safe on empty/missing file.
writeAllLines(file, lines)
Used for cancellations & updates only (full rewrite). Slower but correct for in-place edits.
Sample Record Format
// users.txt (pipe-delimited)
1|Abdul Rehman|rehman@mail.com|03001234567|pass123|2026-01-01 10:00|Lahore|2026-01-01 10:00

// bookings.txt
1|1|FLIGHT|0|3|2026-01-15 09:30|CONFIRMED

// bills.txt
1|1|55.00|2.75|57.75|PENDING

Supporting Design Elements
Enumerations & Custom Exception
Status Enumerations

PENDINGCONFIRMEDCANCELLED
PaymentStatus
PENDINGCOMPLETEDFAILED
PaymentMethod
CASHCARDONLINE
Using enums prevents invalid state strings like "cancelled" vs "CANCELLED" by enforcing compile-time safety instead of runtime string comparison.
SmartGoException
public class SmartGoException
    extends Exception {

  public SmartGoException(String msg) {
    super(msg);
  }
}
Thrown by every service for business rule violations. GUI catches it and shows the message in a Swing error dialog. Keeps error messaging consistent across the whole app.
Where thrown: duplicate email · wrong credentials · flight full · bill already paid · unauthorized cancel · invalid rating
