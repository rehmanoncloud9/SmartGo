Project Motivation
The Problem: Fragmentation in Travel

Travelers currently face a "Logistics Chaos":

Data Silos: Flights, Hotels, and Tours exist in separate systems, making manual planning exhausting.
Billing Leakage: Agencies manually calculate fees (5% rate), leading to massive financial errors.
Inventory Lag: Real-time capacity (seats/rooms) is rarely synced, causing overbooking frustration.
SmartGo's Mission

To build a Unified Object Model where every travel entity is logically linked. By automating the billing pipeline and enforcing security via Encapsulation, we provide a "One-Stop Solution" for modern travel needs.

OOP Pillar 1
Encapsulation: The Data Shield
// From Person.java - Private Shielding
public abstract class Person {
  private int id;
  private String name;
  private String passwordHash;

  // Reader methods (Getters)
  public int getId() { return id; }
  public String getName() { return name; }
}
Technical Rationale
Prevents external classes from modifying sensitive fields like id.
Allows for **Read-Only** access to fields like `passwordHash`.
Ensures that if we change the internal storage format, the rest of the app doesn't break.
Public Access: Allowing any class to set user.id = -1, crashing the database.
VS
Encapsulation: Private fields + Read-only getters. The object is in complete control.

OOP Pillar 3
Inheritance: Code Reuse & Hierarchy
// From Person.java - The DNA
public Person(int id, String name, String email) {
  this.id = id;
  this.name = name;
  this.email = email;
}
// From User.java - Specializing
public User(int id, String name, ..., String addr) {
  super(id, name, email, phone, password);
  this.address = addr;
}

Why use Inheritance?

In SmartGo, we have Users and Admins. Inheritance allows us to write name/email/ID logic once in Person, and every subclass gets it for free. This is the DRY (Don't Repeat Yourself) principle.

SOLID Principles
SRP: Single Responsibility Principle
Package Modularity

Every class has one job. Service classes handle logic, while DataStore handles persistence. This is the definition of SRP.

// Our Modular Package Structure
package services;  // Only logic
package models;    // Only data
package data;      // Only file I/O

SOLID Principles
LSP: Liskov Substitution Principle
Predictable Subclasses

A child class must be 100% compatible with its parent. We treat every account as a Person safely during data loading.

// Safe LSP usage in Loaders
List<Person> allUsers = DataStore.loadUsers();

for(Person p : allUsers) {
  System.out.println(p.getName());
}

SOLID Principles
OCP: Open/Closed Principle
// OCP in Action
public void processBooking(Transport t) {
  // CLOSED for modification
  // OPEN for new transport types
  double price = t.getPrice();
}
Future Proofing

Adding a 'Cruise' booking next year won't require changing a single line of our BookingService. We just extend Transport.
