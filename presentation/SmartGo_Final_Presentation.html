Project Overview
Technical Summary
What Is SmartGo?
The Problem
• Fragmented Systems: Travelers currently manage flights, hotels, and tours across 3+ disconnected applications.
• Manual Calculation: Fee structures and platform commissions are often calculated manually, leading to financial errors.
• Persistence Gaps: Critical user state and booking history are lost between sessions due to lack of local persistence.
The SmartGo Answer
A Unified Java Core where every travel entity is a first-class Object. One codebase handles the entire lifecycle: Discovery → Booking → Billing → Review.
Feature Matrix
Flight Booking
Global search and instant seat reservation.
Hotel Reservations
Rating-based discovery with guest management.
Tour Packages
Multi-day itineraries with optional meal plans.
Automated Billing
5% platform fee computation and persistence.
Review Architecture
Standardized feedback loop across all entities.

OOP Concept 1 of 8
Abstraction
Abstraction — Hiding Implementation Details
// Person.java — abstract base class
public abstract class Person {
  private int id;
  private String name, email, phone, passwordHash;

  // Every subclass MUST implement this
  public abstract String getRole();

  public String toString() {
    return "ID: " + id + " | Role: " + getRole();
  }
}
// Transport.java — abstract with forced contract
public abstract class Transport {
  private double price;
  private int capacity;

  // Subclasses provide their schedule format
  public abstract String getScheduleSummary();
}
Why We Used This Here
Person and Transport are abstract because a plain "Person" or "Transport" has no meaning in our domain.
Abstract classes define template behavior (toString, getPrice) while delegating variable behavior to subclasses.
The compiler prevents new Person() — invalid object creation is impossible.
❌ Alternative: Concrete classes with a "type" String field and runtime checks — messy, error-prone, no compile safety.
VS
✅ Our Choice: Abstract classes give compile-time safety and force correct design.

OOP Concept 2 of 8
Encapsulation
Encapsulation — Controlled Access to Data
// Bill.java — totalAmount is computed internally
public class Bill {
  private double baseAmount, platformFee;
  private double totalAmount; // never set externally
  private String status;

  public Bill(int id, int bookingId,
        double base, double fee) {
    this.baseAmount = base;
    this.platformFee = fee;
    this.totalAmount = base + fee; // auto-computed
    this.status = "UNPAID"; // safe default
  }

  // No setter for totalAmount — read-only
  public double getTotalAmount() { return totalAmount; }
  public void setStatus(String s) { this.status = s; }
}
Why We Used This Here
All fields are private across every model class. External code only reads/modifies what we explicitly allow.
totalAmount has no setter — it can never be tampered with after creation. Prevents billing fraud.
passwordHash in Person is private — AuthService validates it but no other class can read it directly.
Default status = "UNPAID" in constructor ensures objects always start in a valid state.
❌ Without: Public fields mean any class can set bill.totalAmount = 0 — silent fraud, no audit trail.
VS
✅ With: Only BookingService.payBill() can transition status to PAID, after a real payment is recorded.

OOP Concept 3 of 8
Inheritance
Inheritance — Code Reuse via IS-A Relationships
// Flight.java — inherits from Transport
public class Flight extends Transport
    implements Reviewable {

  private String airline, flightNumber, flightClass;
  private String departureTime, returnTime;

  public Flight(int id, int destId,
        double price, int cap, String airline, ...) {
    super(id, destId, "Flight", price, cap);
    this.airline = airline;
  }

  @Override
  public String getScheduleSummary() {
    return "Departs: " + departureTime +
           " | Returns: " + returnTime;
  }
}
Why We Used This Here
Flight IS-A Transport — inherits price, capacity, destinationId, and toString() without rewriting them.
super() delegates base construction to Transport. Flight only initializes its own airline-specific fields.
Admin extends Person and User extends Person — both reuse full authentication and identity infrastructure.
@Override is explicit — prevents typo-based method hiding without a compiler warning.
Inheritance Tree
Person → User    Person → Admin
Transport → Flight  Transport → Bus  Transport → Train
Exception → SmartGoException

    String type, int id) { ... }
// Same method: FLIGHT, HOTEL, TOUR_PLAN
Why We Used This Here
The Reviewable interface lets ReviewService handle Hotel, Flight, and TourPlan uniformly — no switch-case on type needed.
getScheduleSummary() is overridden in 3 concrete classes — the JVM dispatches to the right implementation at runtime.
This is the Open/Closed Principle: adding a new transport type (e.g., Ferry) requires zero changes to BookingService.
❌ Without: if (type.equals("FLIGHT")) {...} else if (type.equals("BUS")) {...} — grows with every new type.
VS
✅ With: transport.getScheduleSummary() — one line works for all transport types today and tomorrow.

Polymorphism — Compile-time
Method Overloading — Same Name, Different Signature
// AuthService.java — two login() overloads

// Login by email + password
public static User login(
    String email, String password)
    throws SmartGoException {
    // find user by email, compare password
}

// Login by userID + password (admin use case)
public static User login(
    int userId, String password)
    throws SmartGoException {
    // find user by ID, compare password
}

// Resolved at COMPILE TIME by parameter types
AuthService.login("email@x.com", "pass"); // → 1st
AuthService.login(42, "pass"); // → 2nd
Why We Used This Here
Compile-time polymorphism allows different calling conventions without renaming the method — callers use the natural login() name.
The compiler selects the right overload based on argument types: String+String → email login; int+String → ID login.
Cleanly separates two distinct login flows without an if-else branch inside a single method.

OOP Concept 5 of 8
Interfaces
Interfaces — Designing Contracts, Not Implementations
// Reviewable.java — a pure contract
public interface Reviewable {
  int getId();
  String getDisplayName();
  String getReviewType();
}

// Hotel implements Reviewable
public class Hotel implements Reviewable {
  @Override
  public String getDisplayName() { return name; }
  @Override
  public String getReviewType() { return "HOTEL"; }
}

// Flight: extends Transport AND implements Reviewable
public class Flight extends Transport
    implements Reviewable { ... }
Why We Used This Here
Java does not support multiple inheritance of classes. But a class can implement multiple interfaces — Flight extends Transport and also implements Reviewable.
Reviewable is a behavioral contract — not about what an object IS, but what it CAN DO (be reviewed).
ReviewService works with any Reviewable now or in future. Zero changes needed to add a new reviewable type.
❌ Abstract class for Reviewable: Forces Hotel to stop extending its own hierarchy just to be reviewable.
VS
✅ Interface: Hotel and Flight keep their own hierarchies AND gain review capability — no trade-off.

OOP Concept 6 of 8
Custom Exceptions
Custom Exception — SmartGoException
// SmartGoException.java
public class SmartGoException extends Exception {
  public SmartGoException(String message) {
    super(message);
  }
}

// Thrown with human-readable messages
throw new SmartGoException(
    "An account with this email already exists."
);

// Caught cleanly in Main.java
try {
  AuthService.login(email, password);
} catch (SmartGoException e) {
  JOptionPane.showMessageDialog(
    null, "Login failed: " + e.getMessage()
  );
}
Why We Used This Here
Domain-specific error type — callers catch SmartGoException, not generic Exception. Our domain errors are distinguishable from Java I/O errors.
Checked exception (extends Exception) — compiler forces every caller to handle or declare it. No silent failure.
Human-readable messages go straight into JOptionPane — user sees "Incorrect password", not a stack trace.
All service methods declare throws SmartGoException.
❌ Without: Return null on failure — null pointer crashes, silent failures, terrible UX.
VS
✅ With: Throw SmartGoException — compiler-enforced handling, user-friendly error display.

OOP Concept 7 of 8
Enumerations
Enums — Type-Safe Constants
// BookingStatus.java
public enum BookingStatus {
  PENDING, CONFIRMED, CANCELLED
}

// PaymentStatus.java
public enum PaymentStatus {
  PENDING, COMPLETED, FAILED
}

// PaymentMethod.java
public enum PaymentMethod {
  CASH, CARD, BANK_TRANSFER
}

// Only valid states allowed — compiler enforces
booking.setStatus(BookingStatus.CANCELLED);

// Serialize: status.name() → "CANCELLED"
// Deserialize: BookingStatus.valueOf("CANCELLED")
Why We Used This Here
Type safety — assigning status = "canceld" (typo) is a compile error when using an enum. String fields have no such protection.
Enums work with switch expressions — exhaustiveness checking prevents missing cases.
Serialization: status.name() writes "CONFIRMED" to file; BookingStatus.valueOf() reads it back — zero parsing boilerplate.
Three separate enums for three separate domains — each concept stays in its own type.
❌ With Strings: "paid", "PAID", "Paid" all pass compilation but break equality checks at runtime.
VS
✅ With Enums: Only .CONFIRMED, .PENDING, .CANCELLED compile. Compiler is the gatekeeper.

File Serialization
File Serialization — Persistent Object State
// DataStore.java — serialize User to pipe-delimited line
public static void saveUser(User user) {
  String line = user.getId() + "|"
    + user.getName() + "|" + user.getEmail()
    + "|" + user.getPasswordHash()
    + "|" + user.getAddress();
  appendLine(USERS_FILE, line);
}

// Deserialize — reconstruct User from line
public static List<User> loadUsers() {
  List<User> users = new ArrayList<>();
  for (String line : readLines(USERS_FILE)) {
    String[] p = line.split("\|");
    if (p.length >= 8)
      users.add(new User(Integer.parseInt(p[0]),
        p[1], p[2], ...));
  }
  return users;
}
Why We Used This Here
State persistence — every User, Booking, Bill, Payment, and Review survives between restarts via pipe-delimited .txt files.
DataStore is a dedicated data-access layer — all read/write logic in one class, keeping models pure and services clean.
Append vs Overwrite: saveUser() appends (fast). saveAllBookings() overwrites entire file (needed to update status mid-list after cancellation).

SOLID Principle
Liskov Substitution
Liskov Substitution — Subclasses Don't Break Contracts
// LSP — Transport reference holds any subtype
// without ANY change in calling code behavior

public static Booking bookFlight(int userId, int flightId) {

  // Flight IS-A Transport — fully substitutable
  Flight flight = FlightService.getFlightById(flightId);

  // getPrice() from Transport — works on ANY subtype
  double fee = flight.getPrice() * PLATFORM_FEE_RATE;

  // Same line works for Bus, Train, etc.
}

// Person reference — User and Admin interchangeable
Person p = AuthService.getLoggedInUser();
System.out.println(p.getRole()); // "User" or "Admin"
What LSP Means in SmartGo
Every subclass of Transport fulfills the full Transport contract — getPrice(), getCapacity(), getScheduleSummary() — without weakening any guarantee.
BookingService calls flight.getPrice() via the Transport reference. Switching to Bus or Train doesn't change the billing logic.
User and Admin are both valid substitutes for Person — neither breaks toString() or any shared logic.
❌ LSP Violation: Subclass overrides getPrice() to throw UnsupportedOperationException — code using Transport reference crashes unexpectedly.
VS
✅ LSP Satisfied: Every concrete Transport subclass maintains the parent's contract — substitution is always safe.

Technical Deep Dive
Clever Pattern — Capturing Console Output for Swing
// Main.java — captureOutput() helper
// Services print to System.out. We redirect
// that output into a String for Swing display.

private static void captureOutput(Runnable action, String title) {

  ByteArrayOutputStream baos = new ByteArrayOutputStream();
  PrintStream ps = new PrintStream(baos);
  PrintStream old = System.out;

  System.setOut(ps);    // redirect stdout
  action.run();        // service prints here
  System.out.flush();
  System.setOut(old);  // restore stdout

  JTextArea ta = new JTextArea(baos.toString());
  ta.setEditable(false);
  JScrollPane sp = new JScrollPane(ta);
  JOptionPane.showMessageDialog(null, sp, title, JOptionPane.PLAIN_MESSAGE);
}
Why This Is Architecturally Sound
Services stay clean — they print to System.out normally. Zero Swing imports in any service class. UI and logic stay fully separated.
Runnable functional interface — Main passes () -> FlightService.showAllFlights() as a lambda. The helper doesn't care what runs inside.
JScrollPane wraps the JTextArea — any amount of output is scrollable, not truncated by a dialog box limit.
stdout is always restored after capture — System.out works normally outside the dialog.
Usage throughout Main.java:
captureOutput(() -> FlightService.showAllFlights(), "All Flights");

Technical Decisions
Security & Data Integrity Choices
🔐 Password Handling
Passwords are stored in passwordHash field (named to signal future hashing intent). BCrypt is the noted production upgrade path.
// AuthService.register()
// Production: BCrypt.hashpw(password, salt)
User newUser = new User(id, name, email, phone, password, ...);
🛡 Ownership Validation
Before cancellation or payment, BookingService checks that the booking's userId matches the logged-in user. Users cannot cancel or pay for other users' bookings.
✅ Input Validation
Email uniqueness enforced at registration — SmartGoException thrown on duplicate.
Review rating validated: must be 1–5. Comment must not be blank.
Payment method limited to enum: CASH, CARD, BANK_TRANSFER — no free-text inputs.
Already-cancelled bookings cannot be cancelled again; already-paid bills cannot be paid again.
💾 Data Consistency
Cancellation loads ALL bookings, modifies the right one, then writes ALL back — atomic update preventing partial writes. Same pattern for bill payments.

Reflection
Key Learnings & OOP Insights
What We Got Right
Abstraction pays off — adding a new booking type (e.g., cruise) requires a new model class + one method in BookingService. Zero existing code changes.
Single DataStore was the right call — centralizing all I/O meant we only debugged file parsing once, not six times.
Custom exceptions dramatically improved UX — every error is a readable English sentence, not a Java stack trace.
Enums prevented bugs — we never had a status-mismatch bug because invalid states simply cannot be compiled.
What We'd Improve
BCrypt password hashing — the passwordHash field name shows intent; production version would use BCrypt.hashpw().
Replace flat files with SQLite — relational queries (JOIN bookings to bills) would be far cleaner than nested Java loops.
Observer pattern for notifications — booking confirmation could trigger email via an event system, decoupled from BookingService.
