Software Requirements Specification
Functional & Non-Functional Requirements
Functional Requirements
FR-01 Guest may Register, Login, or Exit
FR-02 System enforces unique email per account
FR-03 User may search flights by destination city
FR-04 User may book flight, hotel, or tour plan
FR-05 System auto-generates bill with 5% fee on booking
FR-06 User may pay or cancel their own bookings only
FR-07 User may submit star rating + comment review
FR-08 Admin may add/edit/deactivate inventory items
FR-09 All data must persist across application restarts
FR-10 IDs must remain unique across sessions
Non-Functional Requirements
NFR-01 Performance: Menu response < 500ms on standard hardware
NFR-02 Security: User authentication with plain-text password storage
NFR-03 Reliability: Missing data directory auto-created on startup
NFR-04 Usability: All paths reachable within 3 menu selections
NFR-05 Portability: Runs on any JDK 11+ platform; zero dependencies
NFR-06 Constraint: No external DB engine; flat-file persistence only
NFR-07 Single-user: One active session per JVM process (by design)
Known Constraints & Limitations
  • Static loggedInUser field — single-user design; no concurrent sessions
  • No real payment gateway — payment status is toggled manually
  • No input sanitisation beyond email uniqueness check
  • No file-level transaction rollback on partial write failure

Textual Use Cases
UC-04 · Book Flight
Actor:Authenticated User Precondition:User is logged in; flights exist in DataStore Main Flow:
  1. User selects "Flights" from Main Menu
  2. User selects "Search by City" and enters city name
  3. FlightService queries DataStore, returns matching records
  4. User selects a Flight ID and confirms booking
  5. System checks available capacity (> 0)
  6. BookingService creates Booking record, calculates 5% fee
  7. DataStore appends Booking + Bill records to .txt files
  8. System displays success dialog with Bill ID
Alt (Step 5): Capacity = 0 → SmartGoException → error dialog → return to search
Postcondition: Booking CONFIRMED, Bill PENDING in DataStore
UC-07 · Submit Review
Actor:Authenticated User Precondition:User has completed a trip (Booking CONFIRMED) Main Flow:User selects "Post Review" → Enters rating (1-5) and feedback text → ReviewService validates input → DataStore appends Review record to reviews.txt Alt Flow:Rating not 1-5 → System displays error, prompts to re-enter Postcondition:Review is permanently stored and visible to Administrators
UC-01 · Register
Actor:Guest Precondition:Not logged in Main Flow:Guest submits name, email, phone, password → AuthService checks email uniqueness → User record appended to users.txt → User auto-logged in Alt Flow:Email already exists → SmartGoException("Email taken") → return to Guest Menu Postcondition:User account created, session started
UC-06 · Cancel Booking
Actor:Authenticated User Precondition:Booking exists with status CONFIRMED Main Flow:User selects booking ID → BookingService verifies booking.userId == loggedInUser.id → DataStore.writeAllLines() updates status to CANCELLED → Bill status updated to VOID Alt Flow:UserId mismatch → SmartGoException("Unauthorized") thrown → operation rejected Postcondition:Booking status = CANCELLED, Bill = VOID

Applied Throughout the Codebase
OOP Principles
1
Encapsulation
  • All 15 models use private fields with public getters/setters
  • Bill calculations are sealed inside BookingService — UI never computes fees
  • DataStore I/O is private static; services call only the exposed API
  • Password hash exposed via getter only; never re-settable from outside
2
Inheritance
  • Person → User & Admin — shared id, name, email, phone, passwordHash
  • Transport → Flight, Bus, Train — shared price, capacity, destinationId
  • Eliminates duplication: 400+ lines saved by not repeating common fields
  • Subclass constructors call super() with parent parameters
3
Abstraction
  • abstract Person — declares getRole() as blueprint; cannot be instantiated
  • abstract Transport — declares getScheduleSummary(); forces concrete impl
  • Service layer hides all DataStore internals from GUI layer
  • System enforces consistent contract across all subclasses
4
Polymorphism
  • getRole() overridden in User ("User") and Admin ("Admin") — same call, different result
  • ReviewService accepts any targetId — type field dispatches to Flight or Hotel or TourPlan review at runtime
  • getScheduleSummary() returns different string per transport subtype

System Design
Four-Layer Architecture
Layer 1 — Presentation
Java Swing GUI
JOptionPane menus · JTextArea output · captureOutput() bridge · event listeners
↓ calls service methods     ↑ receives results
Layer 2 — Service
Business Logic
AuthService · FlightService · HotelService · TourPlanService · BookingService · ReviewService
↓ reads/writes via DataStore API     ↑ returns parsed objects
Layer 3 — Model
19 POJO Classes
Person / User / Admin · Transport / Flight / Bus / Train · Booking / Bill / Payment · Hotel / HotelBooking · TourPlan / MealPlan / Review / Destination
↓ DataStore serializes models to pipe-delimited strings
Layer 4 — Data
Flat-File Persistence
DataStore.java · appendLine() · readLines() · writeAllLines() · 11 pipe-delimited .txt files
Why This Design
  • GUI never reads files directly — all access goes through services
  • Services never know about Swing — testable independently
  • DataStore is the single I/O choke-point — all persistence in one class
captureOutput() Bridge
Redirects System.out to a JTextArea. Allowed zero changes to CLI logic when migrating to Swing GUI — all print statements work unchanged.

Data Layer
DataStore.java — Single I/O Choke-Point
Core Methods
appendLine(file, data)
Adds new record without rewriting entire file. O(1) write. Uses FileWriter(append=true).
readLines(file)
Returns List<String[]>. Skips blank lines. Splits on "|" delimiter. Safe on empty file.
writeAllLines(file, lines)
Used for updates & cancellations only — rewrites entire file. Slower but correct for in-place edits.
The Pipe Pattern
private static void appendLine(String file, String data) {
  // FileWriter with append=true — no full rewrite
  try (FileWriter fw = new FileWriter(file, true)) {
    fw.write(data + "\n");
  }
}
// Example record in users.txt:
1|Abdul Rehman|rehman@mail.com|03001234567|pass123|address|lastLogin|USER
  • Pipe "|" chosen — zero collision with user-entered text
  • IDs auto-assigned by scanning max existing ID on init

Encountered and Resolved
Challenges & Solutions
Challenge
No Database Engine
Solution
  • 11 pipe-separated .txt files as relation tables
  • Centralised DataStore class — single I/O point
  • appendLine() for inserts · writeAllLines() for updates
  • db.sql prepared for future MySQL migration
Challenge
CLI-to-Swing GUI Migration
Solution
  • Custom captureOutput() utility method
  • System.out redirected to JTextArea via PrintStream bridge
  • Zero changes to service or model logic required
  • All existing print statements rendered correctly in GUI
Challenge
Persistent ID Continuity
Solution
  • On service init, DataStore scans file for max existing ID
  • New records always use maxId + 1
  • Guarantees uniqueness across restarts and partial files
Challenge
Booking Security
Solution
  • Strict ownership check: booking.getUserId() == loggedInUser.getId()
  • SmartGoException thrown on mismatch — operation halted
  • Admin role verified via getRole() before any admin action
